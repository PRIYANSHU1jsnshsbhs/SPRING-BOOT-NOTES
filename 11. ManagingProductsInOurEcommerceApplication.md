# Product Module Notes (Spring Boot E-commerce)

## 1. Module Overview
The Product module handles:
- Product CRUD
- Product search by category and keyword
- Product image upload
- Pagination and sorting
- Validation and custom exceptions

Flow:
1. Controller receives request
2. Service applies business logic
3. Repository queries DB
4. DTO/Response object returned

---

## 2. API Contract (from your implemented code)

| API | Method | Endpoint | Purpose | Request Body | Response |
|---|---|---|---|---|---|
| Add Product | POST | `/api/admin/categories/{categoryId}/product` | Add product to category | `ProductDTO` | `ProductDTO`, 201 |
| Get All Products | GET | `/api/public/products` | Paginated + sorted products | - | `ProductResponse`, 200 |
| Get Products by Category | GET | `/api/public/categories/{categoryId}/products` | Category-filtered list | - | `ProductResponse`, 200 |
| Get Products by Keyword | GET | `/api/public/products/keyword/{keyword}` | Name keyword search | - | `ProductResponse`, 302 (current code) |
| Update Product | PUT | `/api/admin/products/{productId}` | Update fields | `ProductDTO` | `ProductDTO`, 200 |
| Delete Product | DELETE | `/api/admin/products/{productId}` | Delete by id | - | `ProductDTO`, 200 |
| Update Product Image | PUT | `/api/products/{productId}/image` | Upload/replace image | multipart/form-data | `ProductDTO`, 200 |

Notes:
- Current code returns `HttpStatus.FOUND (302)` for keyword search; REST APIs typically use `200 OK`.
- Current code exposes image update route without `/admin` prefix.

## 4.5 Service Implementation
File: `src/main/java/com/ecommerce/project/service/ProductServiceImpl.java`

Core techniques used:
1. Category existence check via `CategoryRepository`
2. Duplicate product check inside category
3. Computed field:
```java
double specialPrice = product.getPrice() - ((product.getDiscount() * 0.01) * product.getPrice());
```
4. Pagination + sorting using:
```java
Sort sortByAndOrder = sortOrder.equalsIgnoreCase("asc")
        ? Sort.by(sortBy).ascending()
        : Sort.by(sortBy).descending();
Pageable pageDetails = PageRequest.of(pageNumber, pageSize, sortByAndOrder);
```
5. Convert entity list to DTO list with `ModelMapper`
6. Throw custom exceptions for not found / empty results
7. File upload delegated to `FileService`

## 4.6 Controller
File: `src/main/java/com/ecommerce/project/controller/ProductController.java`

Controller responsibilities:
- Parse path/query params
- Validate request body (`@Valid`)
- Call service
- Return proper `ResponseEntity`

Pattern example:
```java
@GetMapping("/public/products")
public ResponseEntity<ProductResponse> getAllProducts(
        @RequestParam(name = "pageNumber", defaultValue = AppConstants.PAGE_NUMBER, required = false) Integer pageNumber,
        @RequestParam(name = "pageSize", defaultValue = AppConstants.PAGE_SIZE, required = false) Integer pageSize,
        @RequestParam(name = "sortBy", defaultValue = AppConstants.SORT_PRODUCTS_BY, required = false) String sortBy,
        @RequestParam(name = "sortOrder", defaultValue = AppConstants.SORT_DIR, required = false) String sortOrder
) {
    ProductResponse productResponse = productService.getAllProducts(pageNumber, pageSize, sortBy, sortOrder);
    return new ResponseEntity<>(productResponse, HttpStatus.OK);
}
```

## 9. Important Concepts Usually Missed

## 9.1 Correct HTTP Semantics
- Search/list APIs should return `200 OK`.
- `302 FOUND` is redirect semantics, not normal list response semantics.
- `DELETE` can return either deleted object (current approach) or a plain status response.

Quick fix:
```java
return new ResponseEntity<>(productResponse, HttpStatus.OK);
```

## 9.2 Validation at DTO Boundary
- Put validation on `ProductDTO` so invalid payloads fail before mapping.
- Keep entity validation too, but treat DTO validation as request-contract validation.

Useful annotations:
- `@NotBlank` on `productName`, `description`
- `@PositiveOrZero` on `quantity`, `discount`
- `@Positive` on `price`

## 9.3 Recompute Derived Fields Centrally
- `specialPrice` is a derived field.
- Always compute from `price` and `discount` inside service during create/update.
- Do not trust client-provided `specialPrice`.

Recommended helper:
```java
private double calculateSpecialPrice(double price, double discount) {
    return price - ((discount * 0.01) * price);
}
```

## 9.4 Transaction Boundaries
- Write operations should be transactional (`create`, `update`, `delete`, image update).
- Read operations can be `@Transactional(readOnly = true)`.

Why:
- Prevent partial writes.
- Better consistency under concurrent requests.

## 9.5 Pagination Guardrails
- Validate `pageNumber >= 0`.
- Clamp or validate `pageSize` to a max (example: 100).
- Whitelist sortable fields (`productId`, `productName`, `price`) to avoid invalid column errors.

## 9.6 Search Behavior and Query Design
- `LIKE %keyword%` works for demo scale.
- For large datasets, consider indexes/full-text search.
- Decide case-insensitive search policy early and keep it consistent.

## 9.7 File Upload Security Basics
- Validate content type (`image/jpeg`, `image/png`, etc.).
- Validate max file size.
- Sanitize extension handling.
- Keep random file names (already done) and avoid original names for persistence.

## 9.8 Error Response Standardization
- Keep one consistent error schema for all errors.
- Include: timestamp, status, message, path, validationErrors (if any).

Benefit:
- Frontend integration becomes simpler and predictable.

## 9.9 API Versioning and Documentation
- Add Swagger/OpenAPI (`springdoc-openapi`) for contract visibility.
- Consider versioning strategy (`/api/v1/...`) before public release.

## 9.10 Test Strategy (Minimal but Strong)
- Service tests:
  - Add product success/fail (duplicate, category not found)
  - Update recomputes `specialPrice`
  - Delete not found path
- Controller tests:
  - Validation failures (`400`)
