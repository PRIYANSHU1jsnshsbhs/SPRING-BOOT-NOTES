## 6. Tight Coupling and Loose Coupling

### Definition

Coupling describes how strongly classes depend on each other.

- Tight coupling: one class directly creates or strongly depends on another concrete class
- Loose coupling: one class depends on an abstraction, usually an interface

### Tight Coupling In This Repo

Relevant files:

- `src/main/java/com/tight/coupling/UserManager.java`
- `src/main/java/com/tight/coupling/UserDatabase.java`
- `src/main/java/com/tight/coupling/TightCouplingExample.java`

Key code idea:

```java
public class UserManager {
    private UserDatabase userDatabase = new UserDatabase();

    public String getUserInfo() {
        return userDatabase.getUserDetails();
    }
}
```

Why this is tight coupling:

- `UserManager` is directly tied to `UserDatabase`
- Replacing the data source requires code changes inside `UserManager`
- Testing becomes harder

### Loose Coupling In This Repo

Relevant files:

- `src/main/java/com/loose/coupling/UserDataProvider.java`
- `src/main/java/com/loose/coupling/UserDatabaseProvider.java`
- `src/main/java/com/loose/coupling/WebServiceDataProvider.java`
- `src/main/java/com/loose/coupling/NewDatabaseProvider.java`
- `src/main/java/com/loose/coupling/UserManager.java`
- `src/main/java/com/loose/coupling/LooseCouplingExample.java`

Key code idea:

```java
public interface UserDataProvider {
    String getUserDetails();
}

public class UserManager {
    private UserDataProvider userDataProvider;

    public UserManager(UserDataProvider userDataProvider) {
        this.userDataProvider = userDataProvider;
    }
}
```

Why this is loose coupling:

- `UserManager` depends on `UserDataProvider`, not a fixed implementation
- You can swap database, web service, or another provider easily
- Code is easier to extend and test

### Steps To Implement Loose Coupling

1. Create an interface for the dependency
2. Create multiple classes implementing that interface
3. Make the main class depend on the interface
4. Pass the implementation from outside

## 7. Hands-On: Tight Coupling and Loose Coupling

### Implementation Flow

#### Tight Coupling

1. Create `UserDatabase`
2. Create `UserManager`
3. Instantiate `UserDatabase` inside `UserManager`
4. Call `getUserInfo()`

#### Loose Coupling

1. Create `UserDataProvider` interface
2. Create `UserDatabaseProvider`
3. Create `WebServiceDataProvider`
4. Create `NewDatabaseProvider`
5. Inject one implementation into `UserManager`
6. Call `getUserInfo()`

### Practical Learning

Tight coupling works, but it is rigid.
Loose coupling is better because dependencies can be replaced without rewriting the dependent class.

## 8. Core Concepts of Spring Framework

### Loose Coupling

Spring helps reduce dependency between components.

### Inversion of Control (IoC)

IoC means object creation and lifecycle are managed by Spring instead of your code.

Instead of:

```java
UserManager manager = new UserManager(new UserDatabaseProvider());
```

Spring does:

- create objects
- connect objects
- manage lifecycle

### Dependency Injection (DI)

DI means dependencies are supplied from outside the class.

Common forms:

- Constructor injection
- Setter injection

### Bean

A bean is an object managed by the Spring container.

Examples from this repo:

- `myBean`
- `myCar`
- `carSpecification`
- `userManagerWithUserDataProvider`

## 9. Spring Container and Configuration

### Spring Container

Spring container is responsible for:

- creating beans
- configuring beans
- injecting dependencies
- managing bean lifecycle

### Container Types

- `BeanFactory`
- `ApplicationContext`

This repo uses `ApplicationContext`.

Example:

```java
ApplicationContext context =
        new ClassPathXmlApplicationContext("applicationBeanContext.xml");
```

### Configuration

In this project, configuration is written in XML.

Example:

```xml
<bean id="myBean" class="car.example.bean.MyBean">
    <property name="message" value="I am a first bean"/>
</bean>
```

### Steps To Configure Spring

1. Add Spring dependencies in `pom.xml`
2. Create POJO classes
3. Create XML config in `src/main/resources`
4. Define beans using `<bean>`
5. Load config using `ClassPathXmlApplicationContext`
6. Fetch beans with `getBean()`

## 14. Dependency Injection (DI)

### Definition

DI is a design pattern where dependencies are provided externally rather than created inside the class.

### Benefits

- loose coupling
- better testability
- better maintainability
- easy replacement of implementations

### Types In This Repo

- Constructor injection
- Setter injection

## 15. Constructor Injection

### Relevant Files

- `src/main/java/car/example/constructor/injection/Car.java`
- `src/main/java/car/example/constructor/injection/Specification.java`
- `src/main/java/car/example/constructor/injection/App.java`
- `src/main/resources/applicationConstructorInjection.xml`

### Code Pattern

```java
public class Car {
    private Specification specification;

    public Car(Specification specification) {
        this.specification = specification;
    }
}
```

### XML Pattern

```xml
<bean id="carSpecification" class="car.example.constructor.injection.Specification">
    <property name="make" value="Toyota"/>
    <property name="model" value="Corolla"/>
</bean>

<bean id="myCar" class="car.example.constructor.injection.Car">
    <constructor-arg ref="carSpecification"/>
</bean>
```

### Steps To Implement Constructor Injection

1. Create dependency class like `Specification`
2. Create dependent class like `Car`
3. Add constructor parameter for the dependency
4. Define both beans in XML
5. Use `<constructor-arg ref="..."/>`
6. Load context and fetch the bean

### When To Prefer Constructor Injection

- when dependency is mandatory
- when object should be fully initialized at creation time
- when you want immutable design

### What Changes In IoC

Without IoC:

- your Java code creates and wires objects

With IoC:

- Spring XML creates and wires objects

### XML Example

```xml
<bean id="userDataProvider"
      class="com.ioc.coupling.UserDatabaseProvider"/>

<bean id="userManagerWithUserDataProvider" class="com.ioc.coupling.UserManager">
    <constructor-arg ref="userDataProvider"/>
</bean>
```

### Steps To Implement IoC

1. Create interface for abstraction
2. Create multiple implementations
3. Make service class depend on the interface
4. Register all classes as beans in XML
5. Inject dependency using constructor or setter
6. Get the final bean from Spring container

## 18. Introduction to Autowiring

### Definition

Autowiring means Spring automatically resolves and injects dependencies.

Types covered in this repo:

- by name
- by type
- by constructor

### Why It Helps

- less manual XML wiring
- faster configuration
- cleaner bean setup for simple cases

### Important Caution

Autowiring can fail if:

- bean names do not match
- multiple beans of same type exist
- constructor dependencies are ambiguous

## 19. Autowiring by Name

### Relevant Files

- `src/main/java/com/example/autowire/name/Car.java`
- `src/main/java/com/example/autowire/name/Specification.java`
- `src/main/java/com/example/autowire/name/App.java`
- `src/main/resources/autowireByName.xml`

### How It Works

Spring matches the property name in the class with the bean id in XML.

Code:

```java
private Specification specification;

public void setSpecification(Specification specification) {
    this.specification = specification;
}
```

XML:

```xml
<bean id="specification" class="com.example.autowire.name.Specification">
    <property name="make" value="Toyota"/>
    <property name="model" value="Corolla"/>
</bean>

<bean id="myCar" class="com.example.autowire.name.Car" autowire="byName"/>
```

### Rule

Property name in class:

- `specification`

Bean id in XML:

- `specification`

These must match.

### Steps

1. Create dependency class
2. Create setter in dependent class
3. Make property name meaningful
4. Create bean with matching id
5. Set `autowire="byName"`

## 20. Autowiring by Type

### Relevant Files

- `src/main/java/com/example/autowire/type/Car.java`
- `src/main/java/com/example/autowire/type/Specification.java`
- `src/main/java/com/example/autowire/type/App.java`
- `src/main/resources/autowireByType.xml`

### How It Works

Spring injects dependency based on class type.

Code:

```java
private Specification specification;

public void setSpecification(Specification specification) {
    this.specification = specification;
}
```

XML:

```xml
<bean id="specification1" class="com.example.autowire.type.Specification">
    <property name="make" value="Toyota"/>
    <property name="model" value="Corolla"/>
</bean>

<bean id="myCar" class="com.example.autowire.type.Car" autowire="byType"/>
```

### Rule

There should usually be only one matching bean of that type.

If multiple beans of the same type exist, Spring may throw an ambiguity error.

### Steps

1. Create dependency class
2. Add setter in dependent class
3. Register one bean of the dependency type
4. Set `autowire="byType"`
5. Load context and test

## 21. Autowiring by Constructor

### Relevant Files

- `src/main/java/com/example/autowire/constructor/Car.java`
- `src/main/java/com/example/autowire/constructor/Specification.java`
- `src/main/java/com/example/autowire/constructor/App.java`
- `src/main/resources/autowireByConstructor.xml`

### How It Works

Spring matches constructor parameter type and injects dependency automatically.

Code:

```java
public class Car {
    private Specification specification;

    public Car(Specification specification) {
        this.specification = specification;
    }
}
```

XML:

```xml
<bean id="specification1" class="com.example.autowire.constructor.Specification">
    <property name="make" value="Toyota"/>
    <property name="model" value="Corolla"/>
</bean>

<bean id="myCar" class="com.example.autowire.constructor.Car" autowire="constructor"/>
```

### Rule

Constructor dependency type must be resolvable clearly by Spring.

### Steps

1. Create dependency class
2. Pass dependency through constructor
3. Register dependency bean
4. Set `autowire="constructor"`
5. Load context and fetch main bean

## 22. Quick Comparison

### Tight Coupling vs Loose Coupling

| Topic | Tight Coupling | Loose Coupling |
|---|---|---|
| Dependency | concrete class | interface / abstraction |
| Flexibility | low | high |
| Testing | harder | easier |
| Extensibility | poor | better |

### Constructor Injection vs Setter Injection

| Topic | Constructor Injection | Setter Injection |
|---|---|---|
| Dependency type | mandatory | optional / flexible |
| Initialization | at object creation | after object creation |
| Immutability | better | weaker |
| Preferred for | required dependencies | optional dependencies |

### Autowiring Types

| Type | Matching Rule |
|---|---|
| byName | bean id matches property name |
| byType | bean type matches property type |
| constructor | bean type matches constructor parameter |

## 23. Interview-Style Answers

### What is IoC?

IoC is a principle where object creation and dependency management are handled by the framework instead of application code.

### What is DI?

DI is the process of supplying dependencies to a class from outside rather than creating them inside the class.

### What is a bean?

A bean is an object managed by the Spring container.

### What is the role of `ApplicationContext`?

It loads configuration, creates beans, wires dependencies, and manages the bean lifecycle.

### What is autowiring?

Autowiring is Spring's feature to inject dependencies automatically by name, type, or constructor.
