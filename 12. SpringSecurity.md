## 2) Fast theory recap

### Authentication vs Authorization

- Authentication: verify identity (example: login with username/password).
- Authorization: enforce permissions after login (example: only ADMIN can open admin APIs).

### Security principles used in this section

- Least privilege: grant minimal required access.
- Secure by design: security from the first version, not afterthought.
- Fail-safe defaults: deny by default unless explicitly allowed.
- Input validation + audit logs.
- Patch dependencies regularly.

### Hashing

- Hashing transforms plaintext password into irreversible output.
- bcrypt includes salting automatically.
- Same password can yield different hashes due to random salt.
- Never store plaintext passwords.

### Why JWT

- Stateless API auth (server does not keep session state).
- Token carries identity claims and expiration.
- Sent as:

```http
Authorization: Bearer <token>
```

- Typical JWT lifecycle:
  - User logs in.
  - Server validates credentials.
  - Server issues signed JWT.
  - Client sends JWT on every API call.
  - Server validates JWT and authorizes request.

  username varchar_ignorecase(50) not null primary key,
  password varchar_ignorecase(500) not null,
  enabled boolean not null
);

create table authorities(
  username varchar_ignorecase(50) not null,
  authority varchar_ignorecase(50) not null,
  constraint fk_authorities_users foreign key(username) references users(username)
);

create unique index ix_auth_username on authorities (username, authority);

## 6) Security implementation map (file by file)

## SecurityConfig.java

Purpose:

- Defines request authorization rules.
- Enables method-level security annotations.
- Registers JWT filter in filter chain.
- Forces stateless session mode for token-based auth.
- Configures user store and password encoder.

Key ideas used:

```java
http.authorizeHttpRequests(auth -> auth
    .requestMatchers("/signin", "/h2-console/**").permitAll()
    .anyRequest().authenticated()
);
http.sessionManagement(s -> s.sessionCreationPolicy(SessionCreationPolicy.STATELESS));
http.exceptionHandling(ex -> ex.authenticationEntryPoint(unauthorizedHandler));
http.csrf(csrf -> csrf.disable());
http.addFilterBefore(authenticationJwtTokenFilter(), UsernamePasswordAuthenticationFilter.class);
```

## JwtUtils.java

Purpose:

- Generate JWT from authenticated username.
- Parse token and extract subject (username).
- Validate signature/expiry/format.
- Read Bearer token from Authorization header.

Key methods to keep:

- getJwtFromHeader(HttpServletRequest request)
- generateTokenFromUsername(UserDetails userDetails)
- getUserNameFromJwtToken(String token)
- validateJwtToken(String authToken)

## AuthTokenFilter.java

Purpose:

- Executes once per request.
- Reads JWT from header.
- Validates JWT.
- Loads user details and sets Authentication in SecurityContext.

Key logic:

```java
String jwt = parseJwt(request);
if (jwt != null && jwtUtils.validateJwtToken(jwt)) {
    String username = jwtUtils.getUserNameFromJwtToken(jwt);
    UserDetails userDetails = userDetailsService.loadUserByUsername(username);
    UsernamePasswordAuthenticationToken authentication =
        new UsernamePasswordAuthenticationToken(userDetails, null, userDetails.getAuthorities());
    SecurityContextHolder.getContext().setAuthentication(authentication);
}
```

## AuthEntryPointJwt.java

Purpose:

- Handles unauthorized errors globally.
- Returns consistent JSON 401 response instead of default HTML/login redirect.

## GreetingsController.java

Purpose:

- Public sample endpoint: /hello
- Role-protected endpoints: /user, /admin
- Sign-in endpoint: /signin generates JWT after authentication

Role checks:

```java
@PreAuthorize("hasRole('USER')")
@GetMapping("/user")

@PreAuthorize("hasRole('ADMIN')")
@GetMapping("/admin")
```

## 7) Step-by-step implementation flow (practical)

1. Add dependencies in pom.xml
- Security, Web, JPA, H2, JWT libs.

2. Configure app properties
- DB URL, H2 console, JWT secret, token expiry.

3. Create DB schema for users/authorities
- Add schema.sql for Spring Security JDBC tables.

4. Build SecurityConfig
- Enable method security.
- Set public endpoints.
- Require auth for all others.
- Stateless session policy.
- Register AuthEntryPointJwt + AuthTokenFilter.
- Create PasswordEncoder bean (BCryptPasswordEncoder).
- Configure UserDetailsService (JdbcUserDetailsManager).

5. Seed users (optional for learning)
- Use CommandLineRunner to create test users + roles.

6. Create JWT utility class
- Header parsing, token generation, validation, claim extraction.

7. Create JWT filter
- Parse token, validate, build Authentication, set SecurityContext.

8. Create unauthorized handler
- Return JSON body for 401.

9. Create login request/response DTOs
- LoginRequest(username,password), LoginResponse(username,roles,jwtToken).

10. Build sign-in endpoint
- Authenticate credentials using AuthenticationManager.
- Generate token via JwtUtils.
- Return token + roles.

11. Protect business endpoints
- Use @PreAuthorize or path-based rules.

12. Test with Postman
- POST /signin with credentials.
- Copy token.
- Use Authorization header as Bearer token.
- Call /user and /admin to verify role restrictions.

## 8) Postman testing checklist

1. Sign in:

```http
POST /signin
Content-Type: application/json

{
  "username": "user1",
  "password": "password1"
}
```

2. Use returned JWT:

```http
Authorization: Bearer <jwt-from-signin-response>
```

3. Validate access:
- /hello should work (if configured public).
- /user should work for USER role.
- /admin should fail for USER role and pass for ADMIN role.

4. Negative tests:
- Invalid token -> 401 JSON error.
- Expired token -> 401 JSON error.
- Missing token on protected path -> 401.

## 9) Form login vs Basic vs JWT (quick comparison)

- Form login:
  - Session-based, browser-oriented.
  - Good for server-rendered web apps.

- Basic auth:
  - Sends credentials each request (Base64, not encryption itself).
  - Should be used only over HTTPS.

- JWT:
  - Stateless and scalable for REST APIs.
  - Better for SPA/mobile/microservices.

## 10) Implementation techniques and best practices

- Use BCryptPasswordEncoder for all passwords.
- Keep JWT secret strong and move to environment variables in real projects.
- Keep expiration short and add refresh flow for production.
- Permit only minimum public endpoints.
- Disable CSRF only for stateless APIs (or handle properly for cookie/session flows).
- Add central exception handling for predictable API error format.
- Avoid logging full tokens or sensitive credentials.
- Add tests for:
  - authenticated vs unauthenticated access
  - role-based authorization
  - expired/invalid token behavior

## 14) Default Spring Security behavior (important interview point)

Out-of-the-box (when spring-boot-starter-security is added):

- All endpoints are secured by default.
- Spring creates a default user (`user`) and generated password (printed at startup).
- Default auth is form login + HTTP Basic.
- CSRF is enabled by default.
- Session is stateful by default (not ideal for JWT APIs).

What we changed for API mode:

- Explicitly permitted only selected paths (`/signin`, `/h2-console/**`).
- Switched session policy to stateless.
- Disabled CSRF for token-based API flow.
- Added JWT filter before UsernamePasswordAuthenticationFilter.

## 15) Request flow in this project (internal mechanics)

1. Client calls `POST /signin` with username/password.
2. AuthenticationManager validates credentials via UserDetailsService + PasswordEncoder.
3. On success, JwtUtils signs and returns JWT.
4. Client calls protected API with `Authorization: Bearer <token>`.
5. AuthTokenFilter runs for request and extracts token from header.
6. JwtUtils validates signature + expiry and extracts username.
7. UserDetailsService loads authorities for that username.
8. SecurityContext is populated with authenticated principal.
9. Authorization rules (`@PreAuthorize` and path rules) decide allow/deny.
10. If denied/invalid/missing auth, AuthEntryPointJwt sends JSON 401.

## 16) Common mistakes + fixes (from real Spring Security JWT setups)

- Mistake: weak or non-base64 JWT secret with decoder-based key logic.
  - Fix: use a properly generated base64 secret or change key creation strategy consistently.

- Mistake: forgetting `ROLE_` convention with hasRole.
  - Fix: store authorities as `ROLE_USER`, `ROLE_ADMIN` when using `hasRole('USER')`.

- Mistake: leaving session state enabled while using JWT.
  - Fix: `SessionCreationPolicy.STATELESS`.

- Mistake: placing JWT filter at wrong position.
  - Fix: `addFilterBefore(authTokenFilter, UsernamePasswordAuthenticationFilter.class)`.

- Mistake: exposing H2 console in production.
  - Fix: limit `/h2-console/**` and frame options to development profiles only.

- Mistake: returning 403/HTML unexpectedly for auth failures.
  - Fix: wire AuthenticationEntryPoint for consistent JSON unauthorized responses.

## 17) Cookie-based JWT variant (theory mentioned in class)

If using cookie auth instead of Bearer headers:

1. On login, set JWT in HttpOnly cookie.
2. Browser auto-sends cookie for matching domain/path.
3. Filter reads token from cookie instead of Authorization header.
4. Keep same validation + SecurityContext logic.

Recommended cookie flags:

- HttpOnly = true
- Secure = true (HTTPS only)
- SameSite = Lax or Strict (based on frontend architecture)
- Short max-age + refresh strategy

When using cookies, revisit CSRF strategy instead of always disabling it.

## 18) Minimal production hardening checklist

- Move `spring.app.jwtSecret` to environment/secret manager.
- Add token expiry policy per environment (short in prod).
- Add refresh token endpoint and revocation strategy.
- Add login attempt throttling / lockout.
- Add audit logs for login success/failure and admin access.
- Enforce HTTPS everywhere.
- Add CORS policy explicitly (allow only trusted origins).
- Add integration tests for 401/403 behavior and role matrix.
- Keep dependencies patched (Spring + jjwt updates).
