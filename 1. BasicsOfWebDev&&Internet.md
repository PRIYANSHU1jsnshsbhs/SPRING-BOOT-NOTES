# Basics of Web Development & the Internet

## The Web, Clients, and Servers

- **The Internet** is the global physical network: cables, routers, data centers, and all the infrastructure that connects devices.
- **The Web** (WWW) is the software layer on top of the Internet that lets computers exchange information using HTTP/HTTPS.
- **The Client** is the application that makes requests and displays responses. Examples: web browser, mobile app, or desktop client.
- **The Server** is the machine or software that listens for requests, processes logic, and returns responses. It is usually always on and ready to serve clients.

## APIs & Types of Requests

An **API (Application Programming Interface)** is like a restaurant waiter.

- The client makes a request to the API.
- The API delivers the request to the server or database.
- The API returns the response back to the client.

In Spring Boot, APIs are typically built using `@RestController`.

### HTTP request types and CRUD mapping

| HTTP Request | CRUD Action | Spring Boot Annotation | Purpose |
| --- | --- | --- | --- |
| `POST` | Create | `@PostMapping` | Send new data to the server (e.g. register a user) |
| `GET` | Read | `@GetMapping` | Retrieve data from the server (e.g. view a profile) |
| `PUT` | Update | `@PutMapping` | Update existing data on the server |
| `DELETE` | Delete | `@DeleteMapping` | Remove data from the server |

## REST API Architecture

REST stands for **Representational State Transfer**. It is a set of design rules for creating web services that are easy to use, scalable, and maintainable.

### Key REST principles

- **Stateless**
  - Each request from client to server must contain all information needed to complete the request.
  - The server does not store session state between requests.
  - Example: send an auth token with every request.

- **Client-Server Separation**
  - The frontend UI and backend logic are separated.
  - The client handles presentation while the server handles business logic and data.

- **Uniform Interface**
  - Use standard HTTP methods and predictable URIs.
  - Resources are identified through URLs.

## HTTP vs HTTPS

- **HTTP** stands for HyperText Transfer Protocol.
  - It transfers data in plain text.
  - It is not secure by default.

- **HTTPS** is HTTP over SSL/TLS.
  - Data is encrypted in transit.
  - It protects against eavesdropping and tampering.

## HTTP Status Codes

HTTP status codes are the server's way of telling the client what happened.

- `1xx` — Informational: the request is being processed.
- `2xx` — Success: the request succeeded.
  - `200 OK` — Standard success response.
  - `201 Created` — A new resource was created.
- `3xx` — Redirection: the client must take further action.
- `4xx` — Client error: the request is invalid or cannot be processed.
  - `400 Bad Request` — Request missing required data or malformed.
  - `401 Unauthorized` — Authentication is required or invalid.
  - `404 Not Found` — The requested resource or endpoint does not exist.
- `5xx` — Server error: the server failed while processing a valid request.
  - `500 Internal Server Error` — General server failure.

## Resource, URI, and Sub-Resource

- **Resource**: The entity or object you interact with via the API. Example: `Product`, `User`, `Order`.
- **URI (Uniform Resource Identifier)**: The endpoint used to access the resource. Example: `www.embarkx.com/api/products`.
- **Sub-Resource**: A child resource linked to a parent resource.
  - Example: product reviews for a specific product.
  - URI example: `www.embarkx.com/api/products/{productId}/reviews`.

## Spring Boot Project Structure for an eCommerce Backend

In a typical Spring Boot eCommerce backend, code is organized by responsibility.

```
src/main/java/com/embarkx/ecommerce/
├── model/           # The RESOURCE lives here (e.g., Product.java)
├── controller/      # The URIs and API REQUEST TYPES are defined here (e.g., ProductController.java)
├── service/         # The SERVER's business logic happens here
└── EcommerceApp     # The SERVER runs from here
```

### Responsibilities

- `model/`
  - Contains data entities and resource definitions.
  - Example: `Product.java`, `User.java`, `Order.java`.

- `controller/`
  - Defines request mappings and endpoints.
  - Example: `ProductController.java` with `@GetMapping`, `@PostMapping`, `@PutMapping`, and `@DeleteMapping`.

- `service/`
  - Contains business logic and application rules.
  - This layer handles operations like validation, processing orders, and interacting with repositories.

- `EcommerceApp`
  - The main application class annotated with `@SpringBootApplication`.
  - It boots the Spring server and starts the application.

## Practical Notes for Quiz Success

- Remember the difference between the Internet and the Web.
- Know how HTTP verbs map to CRUD operations.
- Understand REST principles: stateless, client-server, and uniform interface.
- Be able to distinguish HTTP from HTTPS.
- Know the meaning of common HTTP status codes.
- Understand how resources and URIs are defined in a REST API.
- Recognize the Spring Boot folder structure and where each layer belongs.
