# JPA Notes

## 1) JPA in One Line
JPA (Java Persistence API) is the standard way to map Java objects to relational tables.

## Why it matters
- You work with objects instead of raw SQL for common operations
- Spring Data JPA gives you CRUD behavior with very little code
- It fits naturally into Spring Boot applications

## Mental model
Object -> entity -> table

```java
class Category {
    Long categoryId;
    String categoryName;
}
```

- `categoryId` becomes the primary key
- `categoryName` becomes a normal column

---

## 2) Data Layer Architecture
A simple Spring Boot app is usually split into three layers.

- Presentation layer: controllers, request and response handling
- Service layer: business logic and validation
- Data access layer: repositories and database calls

### Request flow
Browser -> Controller -> Service -> Repository -> Database -> Response

### Why this split is useful
- Controllers stay thin
- Business rules stay centralized
- Database code stays isolated

---

## 3) H2 Database
H2 is a lightweight in-memory database used for development, demos, and tests.

### Why use it
- Fast startup
- No installation required
- Easy to reset
- Great for learning JPA

### Typical config
```properties
spring.application.name=app-crud
spring.h2.console.enabled=true
spring.datasource.url=jdbc:h2:mem:test
```

### What this gives you
- An app name
- The H2 console
- An in-memory database named `test`

## 5) Entity Mapping
An entity is a Java class that JPA stores in a table.

### Common annotations
- `@Entity`
- `@Id`
- `@GeneratedValue`
- `@Table`

### Clean example
```java
import jakarta.persistence.Entity;
import jakarta.persistence.GeneratedValue;
import jakarta.persistence.GenerationType;
import jakarta.persistence.Id;
import jakarta.persistence.Table;

@Entity
@Table(name = "customer")
public class Customer {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    private String firstName;
    private String lastName;
    private String email;
    private String phoneNumber;
}
```

### Field checklist for a customer API
- `id`
- `firstName`
- `lastName`
- `email`
- `phoneNumber`

### Naming tip
Keep entity names, field names, and table names predictable. Use `@Table(name = "CUSTOMER")` if you want explicit control over the table name.

## 6) Repository Layer
Spring Data JPA repositories give you standard database operations without writing SQL.

### Repository example
```java
import org.springframework.data.jpa.repository.JpaRepository;

public interface CustomerRepository extends JpaRepository<Customer, Long> {
}
```

### Built-in methods you get
- `save`
- `findById`
- `findAll`
- `deleteById`
- `existsById`

### Main idea
For normal CRUD, you usually do not need custom SQL.

---

## 7) Service Layer
Put business logic in the service, not in the controller.

### Service example
```java
import org.springframework.stereotype.Service;

@Service
public class CustomerService {

    private final CustomerRepository customerRepository;

    public CustomerService(CustomerRepository customerRepository) {
        this.customerRepository = customerRepository;
    }

    public Customer create(Customer customer) {
        return customerRepository.save(customer);
    }
}
```

### Why this is the right place
- Validation and business rules belong here
- Controllers remain simple
- Repository usage stays consistent

---

## 8) Controller Layer
Expose CRUD endpoints through a REST controller.

### Common endpoint set
- `POST /api/customers`
- `GET /api/customers`
- `GET /api/customers/{id}`
- `PUT /api/customers/{id}`
- `DELETE /api/customers/{id}`

### Controller skeleton
```java
import org.springframework.web.bind.annotation.RequestMapping;
import org.springframework.web.bind.annotation.RestController;

@RestController
@RequestMapping("/api/customers")
public class CustomerController {

    private final CustomerService customerService;

    public CustomerController(CustomerService customerService) {
        this.customerService = customerService;
    }
}
```

### Controller rule
Controllers should receive requests, delegate to services, and return responses. Do not put persistence logic here.

## 9) Primary Key Generation
JPA supports multiple ways to generate IDs.

### `AUTO`
Let the provider choose.
```java
@Id
@GeneratedValue(strategy = GenerationType.AUTO)
private Long id;
```

### `IDENTITY`
Database generates the key, usually via auto-increment.
```java
@Id
@GeneratedValue(strategy = GenerationType.IDENTITY)
private Long id;
```

### `SEQUENCE`
Uses a database sequence.
```java
@Id
@GeneratedValue(strategy = GenerationType.SEQUENCE)
private Long id;
```

With an explicit sequence:
```java
@Id
@GeneratedValue(strategy = GenerationType.SEQUENCE, generator = "order_seq")
@SequenceGenerator(name = "order_seq", sequenceName = "order_sequence", allocationSize = 1)
private Long id;
```

### `TABLE`
Uses a separate table to store ID counters.
```java
@Id
@GeneratedValue(strategy = GenerationType.TABLE, generator = "task_gen")
@TableGenerator(
    name = "task_gen",
    table = "id_gen",
    pkColumnName = "gen_key",
    valueColumnName = "gen_value",
    pkColumnValue = "task_id",
    allocationSize = 1
)
private Long id;
```

### `NONE`
You assign the ID manually.

### Practical guidance
- `IDENTITY`: simple and common for H2 and MySQL-style auto increment
- `SEQUENCE`: best when the database supports sequences well
- `TABLE`: flexible but usually slower
- `AUTO`: useful when you do not want to commit to a strategy yet

## 10) Category Persistence Flow
To persist a `Category`, follow this pattern.

1. Create the entity
2. Add `@Entity`
3. Add `@Id` and generation strategy
4. Create `CategoryRepository extends JpaRepository<Category, Long>`
5. Call `categoryRepository.save(category)` inside the service

### Example entity
```java
import jakarta.persistence.Entity;
import jakarta.persistence.GeneratedValue;
import jakarta.persistence.GenerationType;
import jakarta.persistence.Id;

@Entity
public class Category {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long categoryId;

    private String categoryName;
}
```

## 11) CRUD Implementation Steps

### Build it in this order
1. Add the JPA, web, H2, and test dependencies.
2. Configure `application.properties`.
3. Create the entity class.
4. Create the repository interface.
5. Create the service class.
6. Create the controller class.
7. Test the endpoints with curl or Postman.
8. Verify the schema in the H2 console.

### Safe update pattern
```java
Customer customer = customerRepository.findById(id)
    .orElseThrow(() -> new RuntimeException("Customer not found"));

customer.setFirstName(request.getFirstName());
customer.setLastName(request.getLastName());
customer.setEmail(request.getEmail());
customer.setPhoneNumber(request.getPhoneNumber());

return customerRepository.save(customer);
```

### Delete pattern
Check existence first if you want a better error message.

## 12) Optional in Services
Use `Optional` when reading by ID so missing data is handled cleanly.

### Pattern
```java
public Customer getById(Long id) {
    return customerRepository.findById(id)
        .orElseThrow(() -> new RuntimeException("Customer not found"));
}
```

### Benefit
- Makes missing-record handling explicit
- Avoids null-heavy code

## 13) Stale Object State Exception
This exception usually appears when JPA tries to update an entity that is no longer in sync with the database.

### Common causes
- Concurrent updates
- Detached entities being updated incorrectly
- Missing or misused `@Version`
- Saving stale data after the row has changed

### Fixes that usually help
- Fetch the latest entity before updating
- Keep updates inside transactional service methods
- Modify a managed entity and then save it
- Add optimistic locking with `@Version` when needed

## 15) Implementation Techniques That Age Well

### Good habits
- Keep entities small and focused
- Put validation at the boundary
- Avoid exposing entities directly if the API grows
- Keep persistence inside repositories and services
- Write update logic as fetch -> modify -> save

### Recommended error handling
Use explicit exceptions when data is missing instead of returning null.

### Better layering
Controller -> Service -> Repository is the default path for most CRUD apps.
