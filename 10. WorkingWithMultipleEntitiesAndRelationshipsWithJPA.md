# JPA Relationships + Cascading + Fetch Types (Crisp Notes)

## 1) Big Picture
Real-world backends use multiple entities connected through relationships.

In JPA/Hibernate:
- An `@Entity` maps to a DB table.
- Each object instance maps to a table row.
- Relationships are mapped using annotations.

Core relationship types:
- One-to-One (`@OneToOne`)
- One-to-Many / Many-to-One (`@OneToMany`, `@ManyToOne`)
- Many-to-Many (`@ManyToMany`)

---

## 2) Relationship Types in Your Social Media Project

### A) One-to-One: `SocialUser` <-> `SocialProfile`
Use when one user has exactly one profile.

Current mapping idea:
- `SocialUser` owns inverse side with `mappedBy = "user"`
- `SocialProfile` owns FK column (`@JoinColumn`)

```java
// SocialUser.java
@OneToOne(mappedBy = "user", cascade = {CascadeType.REMOVE, CascadeType.PERSIST, CascadeType.MERGE})
private SocialProfile socialProfile;
```

```java
// SocialProfile.java
@OneToOne
@JoinColumn(name = "social_user")
@JsonIgnore
private SocialUser user;
```

When to use 1:1:
- Split optional data from main table
- Separate sensitive profile data
- Keep base entity lean

### B) One-to-Many / Many-to-One: `SocialUser` <-> `Post`
A user can create many posts, each post belongs to one user.

```java
// SocialUser.java
@OneToMany(mappedBy = "socialUser", cascade = {CascadeType.PERSIST, CascadeType.MERGE})
private List<Post> posts = new ArrayList<>();
```

```java
// Post.java
@ManyToOne
@JoinColumn(name = "user_id")
@JsonIgnore
private SocialUser socialUser;
```

### C) Many-to-Many: `SocialUser` <-> `SocialGroup`
A user can be in many groups, and a group can have many users.

```java
// SocialUser.java (owning side)
@ManyToMany(fetch = FetchType.LAZY)
@JoinTable(
    name = "user_group",
    joinColumns = @JoinColumn(name = "user_id"),
    inverseJoinColumns = @JoinColumn(name = "group_id")
)
private Set<SocialGroup> groups = new HashSet<>();
```

```java
// SocialGroup.java (inverse side)
@ManyToMany(mappedBy = "groups")
@JsonIgnore
private Set<SocialUser> socialUsers = new HashSet<>();
```

## 3) Unidirectional vs Bidirectional
- Unidirectional: only one entity has reference to the other.
- Bidirectional: both entities reference each other (`mappedBy` on inverse side).

Use bidirectional only when you actually need traversal both ways.

---

## 4) Circular References and `@JsonIgnore`
When returning entities directly from REST APIs, bidirectional graphs can cause infinite recursion in JSON.

Used in your project:
- `@JsonIgnore` on relation fields like `Post.socialUser`, `SocialGroup.socialUsers`, `SocialProfile.user`.

Better production approach:
- Return DTOs from controllers instead of JPA entities.

---

## 5) Cascading (Most Important Practical Part)

Common cascade types:
- `PERSIST`: saving parent also saves child
- `MERGE`: updating parent merges child updates
- `REMOVE`: deleting parent removes child
- `ALL`: includes all cascade operations

Your project currently uses targeted cascades (good learning setup):
- `SocialUser -> SocialProfile`: `REMOVE, PERSIST, MERGE`
- `SocialUser -> Post`: `PERSIST, MERGE`

When to avoid `CascadeType.ALL`:
- In shared relationships (`ManyToMany`) where deleting one side should not delete shared records.

---

## 6) Fetch Types

What it controls: when related entities are loaded.

- `LAZY`: load relation on-demand
- `EAGER`: load relation immediately with parent

Default JPA fetch types:
- `@OneToMany` -> LAZY
- `@ManyToOne` -> EAGER
- `@ManyToMany` -> LAZY
- `@OneToOne` -> EAGER

In your project:
- `groups` relation is explicitly `LAZY`.

Rule of thumb:
- Prefer LAZY by default.
- Pull required graph via query/join fetch or DTO projection.

## 11) Common Mistakes + Fixes

1. Infinite recursion in JSON
- Fix: `@JsonIgnore`, or better, DTOs.

2. `StaleObjectStateException`
- Usually from detached/stale entity updates or incorrect cascade/update flow.
- Fix: reload current entity in transaction, update managed instance, then save.

3. Broken bidirectional sync
- Fix: use helper methods that set both sides of relation.

4. Unexpected deletes with cascade
- Fix: avoid `CascadeType.ALL` in shared relations like many-to-many.

## 15) Helper Methods for Bidirectional Sync (Important)

Keep both sides in sync from one method call.

```java
// SocialUser.java
public void addPost(Post post) {
  posts.add(post);
  post.setSocialUser(this);
}

public void removePost(Post post) {
  posts.remove(post);
  post.setSocialUser(null);
}

public void joinGroup(SocialGroup group) {
  groups.add(group);
  group.getSocialUsers().add(this);
}

public void leaveGroup(SocialGroup group) {
  groups.remove(group);
  group.getSocialUsers().remove(this);
}
```

Why this matters:
- Prevents inconsistent object graph in memory.
- Reduces stale update and unexpected SQL issues.

## 17) Service Transaction Pattern (Stale-State Safe)

Use managed entity updates inside transaction.

```java
@Transactional
public SocialUser updateProfile(Long userId, String description) {
  SocialUser user = socialUserRepository.findById(userId)
    .orElseThrow(() -> new RuntimeException("User not found"));

  SocialProfile profile = user.getSocialProfile();
  if (profile == null) {
    profile = new SocialProfile();
    user.setSocialProfile(profile);
  }
  profile.setDescription(description);
  return user;
}
```

Key point:
- Fetch current row first.
- Modify managed object.
- Let transaction flush automatically.

## 18) Repository Query Patterns for Fetching

Use join fetch/entity graph when you need child data with parent.

```java
@Query("select u from SocialUser u left join fetch u.posts where u.id = :id")
Optional<SocialUser> findUserWithPosts(@Param("id") Long id);
```

Use cases:
- Avoid lazy initialization exceptions outside transaction.
- Avoid N+1 query problem in read endpoints.
