# Validations and Custom Exceptions in Spring Boot

Based on Modules 105 to 115 (EmbarkX)

## 1. What Validation Means in Spring Boot
Validation ensures incoming request data is correct before business logic runs.

Why it matters:
- Prevents bad data from reaching DB and services
- Gives clear, user-friendly API errors
- Keeps controller and service code cleaner

Common annotations:
- `@NotNull`
- `@NotEmpty`
- `@NotBlank`
- `@Size(min = x, max = y)`
- `@Email`
- `@Min(value)` and `@Max(value)`

## 4. Add Validations in Entity/DTO
Use validation annotations in request model fields.

Current example from `Category`:

```java
@Entity(name = "categories")
@Data
@NoArgsConstructor
@AllArgsConstructor
public class Category {
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long categoryId;

    @NotBlank
    @Size(min = 5, message = "Category name must contain atleast 5 characters")
    private String categoryName;
}
```

### Add more validations (example)

```java
@NotBlank(message = "Category name cannot be blank")
@Size(min = 5, max = 100, message = "Category name must be between 5 and 100 chars")
private String categoryName;
```

## 5. Use `@Valid` in Controller
Validation triggers only when request body is marked with `@Valid`.

```java
@PostMapping("/public/categories")
public ResponseEntity<String> createCategory(@Valid @RequestBody Category category){
    categoryService.createCategory(category);
    return new ResponseEntity<>("Category added successfully", HttpStatus.CREATED);
}

@PutMapping("/public/categories/{categoryId}")
public ResponseEntity<String> updateCategory(@Valid @RequestBody Category category,
                                             @PathVariable Long categoryId){
    categoryService.updateCategory(category, categoryId);
    return new ResponseEntity<>("Category with category id: " + categoryId, HttpStatus.OK);
}
```

## 6. Make Errors User Friendly (Global Exception Handler)
Handle validation errors globally and return field-wise messages.

```java
@ExceptionHandler(MethodArgumentNotValidException.class)
public ResponseEntity<Map<String, String>> myMethodArgumentNotValidException(MethodArgumentNotValidException e) {
    Map<String, String> response = new HashMap<>();
    e.getBindingResult().getAllErrors().forEach(err -> {
        String fieldName = ((FieldError) err).getField();
        String message = err.getDefaultMessage();
        response.put(fieldName, message);
    });
    return new ResponseEntity<>(response, HttpStatus.BAD_REQUEST);
}
```

Result style:

```json
{
  "categoryName": "Category name must contain atleast 5 characters"
}
```

## 7. Custom Exceptions in Spring Boot
Why custom exceptions:
- Separation of concerns
- Reusable exception logic
- Cleaner service methods
- Better control on error response content

### 7.1 ResourceNotFoundException
Used when entity does not exist.

```java
public class ResourceNotFoundException extends RuntimeException {
    public ResourceNotFoundException(String resourceName, String field, Long fieldId) {
        super(String.format("%s not found with %s: %d", resourceName, field, fieldId));
    }
}
```

Service usage:

```java
Category category = categoryRepository.findById(categoryId)
        .orElseThrow(() -> new ResourceNotFoundException("Category", "categoryId", categoryId));
```

### 7.2 APIException
Used for business-rule errors.

```java
public class APIException extends RuntimeException {
    public APIException(String message) {
        super(message);
    }
}
```

Service usage examples:

```java
if (categories.isEmpty())
    throw new APIException("No category created till now.");

if (savedCategory != null)
    throw new APIException("Category with the name " + category.getCategoryName() + " already exists !!!");
```

## 8. Handle Custom Exceptions in Controller Advice
Map each exception type to proper HTTP status.

```java
@ExceptionHandler(ResourceNotFoundException.class)
public ResponseEntity<String> myResourceNotFoundException(ResourceNotFoundException e) {
    return new ResponseEntity<>(e.getMessage(), HttpStatus.NOT_FOUND);
}

@ExceptionHandler(APIException.class)
public ResponseEntity<String> myAPIException(APIException e) {
    return new ResponseEntity<>(e.getMessage(), HttpStatus.BAD_REQUEST);
}
```

## 9. End-to-End Implementation Steps
1. Add validation dependency in `pom.xml` (`spring-boot-starter-validation`).
2. Add Bean Validation annotations in model/DTO fields.
3. Add `@Valid` on controller request body params.
4. Create `ResourceNotFoundException` for missing resources.
5. Create `APIException` for business-rule validation.
6. Throw custom exceptions in service layer where rules fail.
7. Add global exception handler (`@RestControllerAdvice`) for:
   - `MethodArgumentNotValidException` -> 400 with field errors
   - `ResourceNotFoundException` -> 404
   - `APIException` -> 400
8. Keep controllers thin; push checks into service layer.
9. Test with invalid payloads, duplicate names, empty DB, bad IDs.

## 10. Suggested Validation Upgrade (Production-Ready)

- Add unit/integration tests for all validation and exception cases

## 14. ResponseStatusException vs Custom Exceptions
When to use what:
- `ResponseStatusException`: quick, direct error from controller/service for small cases.
- Custom exceptions (`APIException`, `ResourceNotFoundException`): better for reusable business rules and consistency.

Quick example:

```java
throw new ResponseStatusException(HttpStatus.BAD_REQUEST, "Invalid input");
```

Preferred in layered apps:
- Throw custom exceptions in service
- Convert them to HTTP responses in `@RestControllerAdvice`

## 16. Common Mistakes to Avoid
- Forgetting `@Valid` on controller method parameters
- Writing validations in entity/DTO but not handling `MethodArgumentNotValidException`
- Throwing generic `Exception` instead of domain-specific custom exceptions
- Putting business rules in controller instead of service
- Returning inconsistent error payloads across endpoints
