# Spring Boot Notes: Pagination, DTO Pattern, ModelMapper, Sorting, APIResponse

## 1) What is Pagination and How It Works
Pagination means returning data in small chunks (pages) instead of returning all records at once.

Why we use it:
- Better performance
- Smaller response payload
- Better client-side UX (page by page)
- Easy navigation with page number and size

Core terms:
- pageNumber: zero-based page index (0 means first page)
- pageSize: number of items per page
- totalElements: total matching records in DB
- totalPages: how many pages exist
- lastPage: true if current page is last

Flow:
1. Client sends pageNumber and pageSize.
2. Backend builds Pageable using PageRequest.
3. Repository returns Page<Entity>.
4. Backend maps entities to DTOs.
5. Backend returns content + pagination metadata.

## 3) DTO Pattern
DTO (Data Transfer Object) is used to transfer only the data needed between client and server.

Why DTO:
- Avoid exposing entity internals directly
- Control API contract cleanly
- Decouple persistence model from API model
- Easier validation and evolution of request/response formats

Conceptual flow:
1. Client sends request DTO.
2. Server converts DTO to Entity.
3. DB operation happens on Entity.
4. Server converts Entity to response DTO.
5. Client receives DTO response.

In this project:
- Entity: Category
- DTO: CategoryDTO
- Response wrapper: CategoryResponse

## 5) ModelMapper: Intro and Usage
ModelMapper automatically maps fields with matching names.

Used for:
- DTO -> Entity
- Entity -> DTO

Benefits:
- Less boilerplate mapping code
- Faster development

Watch-out:
- Field names should match
- For complex nested objects, custom mapping may be needed

## 8) Pageable and PageRequest
- Pageable: interface describing pagination request.
- PageRequest: concrete implementation to create Pageable.

Code template:
~~~java
Pageable pageDetails = PageRequest.of(pageNumber, pageSize, sortByAndOrder);
Page<Category> categoryPage = categoryRepository.findAll(pageDetails);
List<Category> categories = categoryPage.getContent();
~~~

## 9) Updating Response Structure with Pagination Details
Use CategoryResponse as API payload:

~~~java
CategoryResponse categoryResponse = new CategoryResponse();
categoryResponse.setContent(categoryDTOS);
categoryResponse.setPageNumber(categoryPage.getNumber());
categoryResponse.setPageSize(categoryPage.getSize());
categoryResponse.setTotalElements(categoryPage.getTotalElements());
categoryResponse.setTotalpages(categoryPage.getTotalPages());
categoryResponse.setLastPage(categoryPage.isLast());
~~~

This is better than returning plain list because client gets navigation metadata.

## 18) Validation Best Practice (Important)
Current code keeps validation annotations in Entity (`Category`).

Recommended approach in APIs:
1. Put request validation constraints in DTO (`CategoryDTO`) to validate incoming payload.
2. Keep Entity focused on persistence concerns.
3. Use `@Valid` in controller for request body.

Suggested DTO shape:
~~~java
public class CategoryDTO {
    private Long categoryId;

    @NotBlank
    @Size(min = 5, message = "Category name must contain atleast 5 characters")
    private String categoryName;
}
~~~

## 20) Common Pitfalls and Fixes
1. Pitfall: returning entities directly.
- Fix: always return DTOs.

2. Pitfall: no default pagination values.
- Fix: define defaults in constants.

3. Pitfall: not handling empty result sets.
- Fix: either return empty content with metadata or throw APIException based on product requirement.

4. Pitfall: invalid sortBy causes runtime error.
- Fix: validate `sortBy` against allowed fields (`categoryId`, `categoryName`).

5. Pitfall: poor error consistency.
- Fix: centralize via global exception handler.
